### Zone/Space model — design locked (Parcel Z, not yet built)

Extended design conversation resolved the zone/space architecture. Key decisions:

**Naming:** "Rooms" renamed to "Spaces" — applies to both indoor rooms (Kitchen) and outdoor areas (Outdoor Kitchen, Pool Area, Garden Bed). Unified entity, context-aware label in UI.

**Zone types on the property:**
- Buildings: Main House, Guest House, White Barn, Red Barn
- Outdoor areas: Yard, Home Perimeter, Near Fields, Far Fields
- Both types support the same functionality; buildings get optional 3D interior view (nice-to-have, no schema impact)

**Zones must become a proper DB table.** Currently zones are JSONB inside `property_visual_config.site_config`. This blocks FK linkage from other entities. Migration needed: extract zones to a `zones` table with UUID PKs; update `spaces` (rooms) table to use real UUID FK instead of string zone_id.

**Project/task/asset linkage — three levels:**
- Space-level (e.g., Kitchen Renovation)
- Zone-level (e.g., Automate Lighting in Main House — applies to whole zone)
- Property-level (e.g., Home Security — applies to full property)
- New Space can be prompted during project creation to avoid completion gaps
- `projects` and `tasks` both get nullable `zone_id` + `space_id` FKs independently — a task can be at a different level than its parent project
- `assets` get nullable `zone_id` + `space_id` (always somewhere specific, not property-level)

**Completion % model — hierarchical, bottom-up only:**
- Space % = items explicitly assigned to that space (projects + tasks + ongoing task instances in active window)
- Zone % = zone-direct items + all space items within that zone
- Property % = all items at any level
- Property-level items do NOT pull down zone/space %
- Zone-level items do NOT pull down space %
- Color coding: 0% red, partial orange/yellow (by %), 100% green

**Ongoing task annual cycle model:**
- `OngoingTask` stays as the template (recurrence, active_months, zone/space/asset linkage)
- New `ongoing_task_instances` table: one row per task per year, with year + status + completed_at
- Tasks only count toward completion % when their active window is open (based on active_months)
- 1/1 annual reset: prior year instances become the archive; new year starts fresh
- Aligns with existing quarterly/annual budget framework

**Maintenance cost aggregation:**
- Asset and zone/space maintenance costs (e.g., mower oil ~$50/yr) should aggregate upward into `global_commitments` as recurring annual commitments
- Flow is owner-curated (or agent-suggested + owner-confirmed), not automatic
- `global_commitments` recurrence_type handles this path — no source FK needed for now
